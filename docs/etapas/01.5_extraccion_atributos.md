# Etapa 1.5 - Extraccion semantica de atributos

**Posicion en el pipeline:** despues de Etapa 1 y antes de construir grafos.

**Objetivo:** convertir `product_name` en columnas estructuradas que sirvan para
grafos semanticos, busquedas BFS y analisis por familias de producto.

Esta etapa no debe repetir el hardcodeo del proyecto base. La regla correcta es:
primero algoritmos deterministas, luego reglas configurables, y finalmente LLM
solo como asesor de reglas si hace falta.

## Entrada

Archivos esperados desde Etapa 1:

- `products_clean.csv`
- `product_activity_summary.csv`
- `code_pattern_summary.csv`
- opcional: un catalogo validado por humanos para evaluar errores.

En este repositorio, si existe `datos/catalogo.csv`, se usa como benchmark parcial
contra la version anterior del proyecto.

## Salida

La etapa genera:

- `product_attributes.csv`: atributos extraidos por producto.
- `attribute_coverage_report.csv`: cobertura por atributo.
- `attribute_review_sample.csv`: muestra priorizada para revision humana.
- `token_profile.csv`: tokens frecuentes del catalogo.
- `attribute_rules.json`: reglas utilizadas o sugeridas.
- `attribute_gold_comparison_summary.csv`: comparacion contra catalogo validado, si existe.
- `attribute_gold_comparison_detail.csv`: detalle producto por producto de esa comparacion.

## Principio de diseno

El LLM no debe modificar codigo ni ejecutar Python. Si se usa, solo puede devolver
un JSON de reglas dentro de un conjunto seguro de operadores:

- `keyword_first_match`
- `keyword_any`
- `contains_any`
- `not_contains_any`

Reglas especificas como `codigo >= 5000`, excluir `VIDRIO`, o asumir una familia
por un patron interno del ERP nunca se aplican automaticamente. Esas reglas pueden
aparecer como candidatas, pero deben validarse con datos, actividad comercial y
revision humana.

## Pipeline

```text
products_clean.csv
    |
    +-- 1. Extraccion universal sin LLM
    |      capacidad, dimensiones, color, material, boca/rosca, tokens
    |
    +-- 2. Perfilado estadistico
    |      frecuencia de tokens, cobertura y patrones raros
    |
    +-- 3. Reglas de dominio
    |      baseline deterministico ahora; LLM opcional despues
    |
    +-- 4. Motor de reglas seguro
    |      aplica JSON/configuracion, no codigo generado
    |
    +-- 5. Validacion
    |      cobertura, confianza, contradicciones y benchmark si existe
    |
    +-- 6. Revision humana
           muestra priorizada por baja confianza y alta actividad
```

## Atributos objetivo

`product_attributes.csv` contiene:

- `product_id`
- `product_name`
- `product_type`
- `subtype`
- `accessory`
- `shape`
- `feature`
- `material`
- `color`
- `capacity_value`
- `capacity_unit`
- `mouth_size_mm`
- `size_text`
- `closure_type`
- `brand`
- `use_category`
- `material_family`
- `keywords`
- `confidence`
- `extraction_method`
- `attribute_evidence`

Cada fila debe ser trazable: `attribute_evidence` indica que token o patron disparo
cada atributo.

## Evaluacion

Hay tres niveles de evaluacion.

1. **Cobertura**
   - Cuantos productos reciben `product_type`, `material`, `color`, etc.
   - No todos los atributos deben cubrir 100%; por ejemplo, no todo producto tiene
     volumen o color explicito.

2. **Consistencia**
   - Ejemplo: `product_type=tapa` con `capacity_value` puede ser sospechoso.
   - Productos con alta actividad y baja confianza pasan a revision humana.

3. **Benchmark**
   - Si existe un catalogo validado, se compara atributo por atributo.
   - En el caso Vidra, `datos/catalogo.csv` permite medir una primera aproximacion
     contra columnas antiguas como `tipo_producto`, `material_principal`,
     `volumen_ml`, `tam_boca_mm` y `uso_categoria`.

## Criterios minimos

Para avanzar a grafos semanticos, el baseline deberia lograr:

- `product_type` con cobertura mayor o igual a 70% en catalogos descriptivos.
- `confidence` promedio mayor o igual a 0.75.
- una muestra de revision humana exportada.
- reglas persistidas en `attribute_rules.json`.

Para una version con LLM, el objetivo sube:

- `product_type` mayor o igual a 90% si el rubro tiene descripciones densas.
- errores medidos contra benchmark o revision humana.
- maximo 1 a 3 llamadas LLM por empresa.

## Reglas configurables por dominio

El codigo no se modifica a si mismo. La parte configurable vive en:

```text
attribute_rules.json
```

Ese JSON puede ser generado manualmente, por analisis de tokens o por LLM. El
motor solo acepta metodos seguros y un esquema limitado:

- `keyword_first_match`
- `keyword_any`
- `contains_any`
- `not_contains_any`

Flujo recomendado:

```text
algoritmo base -> LLM propone JSON -> algoritmo valida y aplica JSON
```

El LLM no ejecuta codigo ni edita archivos Python. Solo propone reglas como:

```json
{
  "attribute_rules": {
    "subtype": {
      "method": "keyword_first_match",
      "rules": [
        {"keywords": ["ESMALTE"], "value": "esmalte", "confidence": 0.92}
      ]
    },
    "accessory": {
      "method": "keyword_any",
      "rules": [
        {"keywords": ["BROCHA"], "value": "brocha", "confidence": 0.92}
      ]
    }
  }
}
```

Luego el pipeline valida el JSON, descarta reglas fuera de esquema y regenera
`product_attributes.csv`.

## Prueba opcional con Gemini

Google recomienda instalar el SDK `google-genai` y crear una API key en AI
Studio. El cliente Python puede usar una API key desde el entorno.

El runner de Gemini usa por defecto todo el catalogo disponible y prioriza, en
el orden del prompt, productos con baja estructura, baja confianza y
descripciones largas. No hace falta configurar `LLM_SAMPLE_SIZE` o
`LLM_TOKEN_LIMIT` para la prueba normal.

```powershell
py -m pip install google-genai
$env:GEMINI_API_KEY="TU_API_KEY"
$env:GEMINI_MODEL="gemini-2.5-flash"
py TF-Final\etapa1_5_atributos_llm_gemini.py
```

Si se quiere reducir el costo de una prueba posterior:

```powershell
$env:LLM_SAMPLE_SIZE="300"
py TF-Final\etapa1_5_atributos_llm_gemini.py
```

Esto genera:

```text
TF-Final/outputs/stage15_datos/attribute_rules_gemini_additions.json
TF-Final/outputs/stage15_datos/attribute_rules_gemini_merged.json
```

Para probar el resultado de Gemini sin reemplazar reglas base:

```powershell
$env:LLM_RULES_PATH="TF-Final\outputs\stage15_datos\attribute_rules_gemini_merged.json"
py TF-Final\etapa1_5_atributos_llm_test.py
```

Esto escribe la comparacion en:

```text
TF-Final/outputs/stage15_datos_llm_test_gemini/
```

Ademas exporta:

```text
TF-Final/outputs/stage15_datos_llm_test_gemini/attribute_llm_delta.csv
```

Ese archivo muestra atributo por atributo que cambio frente al baseline
deterministico.

## Finalizacion automatica

El usuario final no revisa reglas. El sistema aplica una politica conservadora:
Gemini solo se acepta si no empeora metricas criticas y mejora cobertura o
recuperabilidad semantica.

```powershell
py TF-Final\etapa1_5_atributos_llm_finalize.py
```

Si Gemini pasa la politica, se genera:

```text
TF-Final/outputs/stage15_final/
```

Esta carpeta contiene los atributos aceptados, reglas, delta contra baseline,
reporte de recuperabilidad y resumen de decision:

```text
product_attributes.csv
attribute_rules.json
attribute_llm_delta.csv
semantic_retrievability_report.csv
semantic_retrievability_summary.json
llm_acceptance_summary.json
```

Si Gemini no pasa, se genera `outputs/stage15_final_rejected/` con evidencia de
rechazo y el baseline sigue siendo el estado utilizable.
